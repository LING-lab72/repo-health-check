## 5 分钟讲解脚本

### 1. 项目介绍，约 30 秒

大家好，我们的项目叫 Repo Health Check，是一个开源仓库健康度体检工具。

它解决的问题是：很多开发者在选择或维护开源项目时，只能看 Star、README 或主观印象，很难快速判断这个仓库的工程质量、测试情况、文档完善度、安全风险和持续集成情况。

所以我们做了一个可以输入 GitHub 仓库地址，然后自动克隆、分析、评分、生成报告和改进建议的系统。目标用户包括课程项目团队、开源项目维护者，以及想快速评估仓库质量的开发者。

### 2. AI 软件工程实践，约 1 分钟

这个项目不是单纯手写完成的，而是结合了 AI 软件工程流程。

我们使用了 AI 辅助需求拆解、代码审查、安全问题发现、测试补全、前端页面优化和答辩材料生成。比如在后期检查中，AI 帮我们发现了 XSS 风险、Session 密钥硬编码、缓存浅拷贝、对比接口串行等问题，然后我们逐项修复。

和传统开发相比，AI 的提升主要体现在三个方面：

第一，问题发现更快，可以从安全、架构、体验、工程化多个角度一起审查。

第二，迭代速度更快，比如 PDF 导出、分享卡片、语言分布饼图、排行榜分页、CI/CD 都是在迭代中快速补齐的。

第三，项目文档和演示材料也能同步生成，减少了只写代码、不沉淀工程文档的问题。

### 3. Harness / SDD 与工程管理，约 1 分钟

在工程管理上，我们使用了 SDD，也就是规格驱动开发。项目里有 `openspec` 和 `sdd` 目录，用来记录需求、任务拆解和变更说明。

每个需求不是直接开写，而是先拆成可以验证的小任务，比如安全修复、缓存优化、报告增强、PDF 导出、分享卡片、CI/CD 等。

CI/CD 方面，我们配置了 GitHub Actions，同时也适配了 GitLink/Jianmu 流水线。流水线会执行后端测试、前端测试、lint 和 build，保证每次提交不会破坏核心功能。

团队协作上，我们把需求、实现、测试、文档、演示分开管理，这样即使项目后期功能越来越多，也能保持可维护性。

### 4. 产品 Demo 与运营成果，约 1.5 分钟

下面进入 Live Demo。

首先我在首页输入一个仓库地址：

```text
https://github.com/LING-lab72/repo-health-check
```

点击开始分析后，系统会显示分步进度：检查缓存、克隆仓库、分析代码、AI 诊断、生成报告。

进入报告页后，可以看到总分、等级、六维雷达图、语言分布、各维度得分、AI 诊断建议和问题列表。

这里的雷达图有中心向外发散的动画，表示从仓库核心指标向外扩展分析维度。

接着我演示导出能力：点击“导出 PDF”，系统会把当前报告视觉导出为 PDF，而不是只生成简单表格。

然后演示分享能力：点击“生成分享卡片”，可以生成适合发到社交媒体或项目展示页的图片。

最后演示仓库对比：点击“与另一个仓库对比”，输入：

```text
https://github.com/vuejs/vue
```

和：

```text
https://github.com/facebook/react
```

系统会并行分析两个仓库，展示分数对比、维度差异和 PK 结果。

### 5. 经验总结，约 1 分钟

这个项目最大的价值，是把“开源项目质量”从主观判断变成了可量化、可解释、可导出的报告。

过程中遇到的问题主要有三个：

第一，真实仓库差异很大，分析逻辑要兼容 Python、前端、多语言项目。

第二，AI 诊断不能强依赖 API Key，所以我们做了无 Key 的本地规则兜底，有 Key 时自动升级到 DeepSeek 诊断。

第三，工程化细节很多，比如 CORS、端口、启动脚本、GitLink 流水线镜像源、PDF 样式一致性等，都是后期真实测试中暴露出来并修复的。

下一步计划是继续增强社区活跃度分析，比如 Issue 响应时间、PR 合并速度、贡献者多样性；同时加入更多同类项目基准线，让报告不只是看自己的分数，也能知道在同类仓库中的位置。

## 现场网络慢时的兜底话术

如果分析时间比较长，可以这样说：

“因为系统会真实克隆远程 GitHub 仓库并进行多维度扫描，所以第一次分析会受网络和仓库大小影响。为了解决这个问题，我们已经做了 30 分钟缓存机制，同一个仓库重复访问会直接读取缓存结果，避免每次都重新克隆分析。”

如果 GitHub 访问失败，可以这样说：

“这里主要受现场网络影响。系统本身支持错误提示和缓存复用，后端也提供本地自分析接口用于离线演示。我们可以先看已经生成好的报告结果。”
