# 需求规格说明书

## 1. 引言

### 1.1 编写目的

本文档定义 Repo Health Check 的业务目标、用户角色、功能需求、非功能需求、约束条件和验收标准，为开发、测试、答辩和后续维护提供统一依据。

### 1.2 项目背景

开源项目数量庞大，但普通用户、学生开发者和项目维护者往往难以快速判断一个仓库是否健康。传统判断方式依赖人工查看 README、测试目录、依赖文件、Issue 状态和代码结构，效率低且主观性强。Repo Health Check 希望通过自动化分析和可视化报告，降低开源项目评估门槛。

### 1.3 目标用户

| 用户角色 | 典型需求 |
| --- | --- |
| 开源项目维护者 | 快速发现项目质量短板，获得改进建议 |
| 课程评审/教师 | 通过报告、CI、文档和演示判断项目工程化程度 |
| 学生开发者 | 学习开源项目质量指标和软件工程实践 |
| 普通 GitHub 用户 | 判断一个仓库是否值得使用、学习或贡献 |

## 2. 总体目标

1. 输入 GitHub 仓库 URL 后自动生成健康度报告。
2. 从六个维度量化仓库质量。
3. 输出可视化报告、Badge、AI 诊断和导出文件。
4. 支持仓库排行榜、投票、历史趋势和双仓库对比。
5. 通过 OpenSpec、CI/CD、测试和文档体现完整软件工程实践。

## 3. 功能需求

### FR-01 仓库分析

系统应支持用户输入合法 GitHub 仓库 URL，并对仓库进行克隆、扫描、聚合评分。

验收标准：
- 非法 URL 显示输入错误。
- 合法 URL 能进入报告页。
- 分析成功后返回总分、等级、六维分数和问题列表。

### FR-02 六维健康度评分

系统应从以下维度分析仓库：

| 维度 | 内容 |
| --- | --- |
| 代码质量 | 复杂度、重复/坏味道、语言分布、静态检查 |
| 测试覆盖 | 测试文件、测试框架、CI 测试命令 |
| 架构健康 | 目录结构、模块边界、循环依赖、规模风险 |
| 文档完整 | README、贡献指南、变更日志、安全策略 |
| 依赖安全 | 依赖文件、漏洞扫描、锁文件、过期风险 |
| 工程规范 | CI/CD、格式化、Lint、EditorConfig、环境样例 |

### FR-03 报告展示

报告页应展示：
- 总健康分与等级。
- 六维雷达图。
- 维度分数条。
- 语言分布饼图。
- AI 诊断建议。
- 问题列表。
- Badge 嵌入代码。
- 分享链接。

### FR-04 AI 诊断

系统应支持两级诊断：
- 无 API Key：使用本地规则自动生成建议。
- 有 DeepSeek/OpenAI Key：调用 LLM 生成更详细建议。

验收标准：
- 用户不配置 Key 也能获得诊断。
- 配置 Key 后不持久化保存到后端。
- 诊断项包含严重程度、预估工时、置信度和是否需要人工确认。

### FR-05 缓存感知分析

系统应在 30 分钟缓存有效期内直接返回已有结果，避免重复克隆和分析。

验收标准：
- 同一仓库短时间重复访问优先命中缓存。
- 错误缓存不会长期阻塞重新分析。

### FR-06 导出与分享

系统应支持：
- HTML 报告导出。
- 浏览器打印为 PDF。
- 分享卡片生成。
- README Badge 生成。

### FR-07 仓库对比

系统应支持输入两个不同仓库 URL，展示双方总分、维度对比、雷达图、PK Arena 和海报生成。

验收标准：
- 不允许同一仓库自我对比。
- 从报告页跳转时可预填仓库 A。
- A / VS / B 布局稳定居中。

### FR-08 排行榜与投票

系统应记录分析历史，支持排行榜展示、分页、趋势标记和用户投票。

### FR-09 用户认证

系统应支持 GitHub OAuth 登录，用于扩展个性化使用场景。

### FR-10 工程化与文档

系统应包含：
- GitHub Actions CI。
- GitLink DevOps 配置说明。
- OpenSpec capability 归档。
- CONTRIBUTING、SECURITY、CHANGELOG。
- 软件工程文档集。

## 4. 非功能需求

| 类型 | 需求 |
| --- | --- |
| 性能 | 普通中小型仓库 30-180 秒内完成分析；缓存命中应秒级返回 |
| 可用性 | 前端提供进度提示、错误提示、输入校验和一键启动脚本 |
| 安全性 | 防止 XSS、Session 密钥硬编码、OAuth state 缺失、跨域误配 |
| 可靠性 | 后台任务有锁保护，临时目录最终清理，错误缓存可恢复 |
| 可维护性 | 模块按 routes、services、analyzer、components、pages 划分 |
| 可测试性 | 后端 pytest，前端 Vitest，CI 执行 lint/test/build |
| 可扩展性 | 可继续接入更多分析维度、数据库、CI 平台和语言分析器 |

## 5. 约束条件

1. GitHub 仓库需要可通过网络克隆。
2. 大型仓库可能超时，需要提示用户稍后重试。
3. LLM 诊断依赖用户提供或环境配置 API Key。
4. Windows 本地运行需安装 Python、Node.js 和 Git。
5. GitHub/GitLink CI 依赖平台网络和构建镜像。

## 6. 验收标准

1. 双击 `Start-RepoHealth.bat` 可启动前后端。
2. 首页高级 UI、Dock 导航和报告页正常展示。
3. 输入公开 GitHub 仓库可生成报告。
4. Compare、Leaderboard、About 页面功能可访问。
5. `npm run lint`、`npm test -- --run`、`npm run build` 通过。
6. 后端 pytest 和 E2E 脚本可执行。
7. GitHub 上存在稳定版本标签 `v2.1.0`。
